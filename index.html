In the era of AI code generation, code volume exceeds human review capacity. A single malicious line hidden in a large PR is nearly impossible to find. Topo's .topo declarations create an auditable machine-verifiable contract that reduces review to three things: the declarations, any external functions, and trust in the checker.

Overview

What is Topo

Topo extracts the logical requirements of a program — what's public, what must run before what, what can run in parallel — from implicit code patterns into explicit .topo declarations.

Undeclared Behavior Is Forbidden

Non-external functions cannot access I/O, networking, or process capabilities. Violations are compile-time errors, not warnings buried in a log.

Declarations Drive Optimization

Explicit stage ordering, parallelism, and visibility give the compiler information beyond what traditional static analysis can derive — enabling inlining, reordering, and auto-parallelization.

Review the Declarations, Not the Diff

A 1000-line PR might have 3 unsafe points marked external. Reviewers check those and the .topo changes — the rest is machine-verified.

Declaration Types

Three Declaration Classes

Every .topo declaration falls into one of three classes. Together they describe the full logical structure of a program.

Boundary

Existence Boundary

Which symbols are public API, which are implementation detail, and which are internal to a stage. Enables visibility verification, dead code elimination, and symbol obfuscation.

public:
protected:
private:
internal:
Order

Logic Ordering

Which operations must precede which via stage<N> annotation. Enables stage-isolation verification and safe instruction reordering across stage boundaries.

stage<1> init();
stage<2> process();
stage<3> finalize();
Parallel

Logic Parallelism

Which operations are independent at the same stage. Functions at identical stage numbers form a pipeline DAG, enabling automatic parallelization and concurrent code generation.

stage<2> physics::step();
stage<2> tick_ai(); // independent
stage<2> mix_audio(); // all three run in parallel

Get Started

Quick Start

Three steps from zero to verified build. Works with C++, Rust, Java, Python, and TypeScript.

1

Write Declarations

Create a .topo file describing your program's logical structure.

namespace app {
public:
int process(int x);
protected:
int helper(int x);
}
2

Configure

Create a Topo.toml to tell the toolchain about your project.

[project]
name = "my-project"

[topo]
root = "topo/main.topo"

[build]
language = "cpp"
sources = ["src/*.cpp"]
3

Check and Build

Run the checker, then build with optimization informed by your declarations.

$ topo-check .
✓ All checks passed

$ topo-build
Build succeeded

Requires CMake 3.20+, Ninja, and vcpkg. LLVM is auto-managed by the build system.

git clone https://github.com/Topo-lang/topo-core.git && cd topo-core && cmake --preset default && cmake --build build

Safety

Three-Layer Safety Verification

Every host language gets check. Each layer catches a different class of violation.

L1

Regex Scanning

Pattern-based detection of direct API calls, I/O operations, and escape patterns. Runs on any host language with zero dependencies — no toolchain SDK required.

L2

Semantic Analysis

Resolved-symbol analysis through LSP bridges (clangd, rust-analyzer, tsserver, jdt). Catches indirect capability use, macro expansion, and template-instantiated violations.

L3

Post-Transform Verification

Linkage and visibility re-verification after backend optimization passes. Ensures that transforms preserved the declaration contract through every pipeline stage.

Optimization

Measured Optimization Impact

The same .topo declarations that power check tell the compiler what's safe to transform. Results from Topo's 55-benchmark suite (Apple M1, forced-mode on friendly workloads).

3.3×

JVM Auto-Parallelization

Same-stage functions proven independent by .topo pipeline DAG. Backend emits concurrent bytecode with lock elision — safety guaranteed by the declaration, not by runtime checks.

JVM — Java
2.1×

LLVM Data-Layout Transform

Visibility and stage ordering prove strided access patterns. Compiler switches struct-of-arrays to array-of-structs layout — streaming workloads benefit from contiguous memory access.

LLVM — C++ / Rust
1.9×

LLVM Arena Allocation

Visibility declarations prove bounded object lifetime. Compiler replaces heap allocation with arena — no malloc/free, no GC pressure, cache-warm reuse within the declared scope.

LLVM — C++ / Rust
